# Dependency Inversion Principle

- DIP는 high level 모듈들이 low level 모듈들과 의존 관계를 맺으면 안 된다는 원칙으로, 둘의 의존 관계는 abstraction을 통해 이뤄져야 한다는 원칙이다.
- abstraction은 구체적인 구현 클래스에 의존해서는 안되고, 구체적인 구현 클래스가 abstraction에 의존해야 한다.

- IoC 원칙을 따르기 위해 비지니스 로직 클래스에서 데이터 접근 클래스의 인스턴스를 팩토리 클래스 메소드로부터 가져온다고 하더라도 만약 팩토리 클래스의 메소드가 데이터 접근 클래스의  인터페이스 클래스가 아니라 concrete class를 반환한다면, 비지니스 로직 클래스와 데이터 접근 클래스는 여전히 tight coupling을 형성하고 있는 것이다. 따라서 두 클래스는 abstraction을 통해 의존 관계를 형성할 필요가 있다.
- DIP에서 high level module과 low level module
  - high level module : high level 모듈이란 다른 모듈에 의존하여 메소드를 구현하는 모듈이다.
  - n-tier 아키텍쳐에서는 비지니스 로직 모듈이 데이터 접근 모듈에 의존하여 비지니스 로직 메소드들을 구현하고 있으므로 비지니스 로직 모듈은 high level 모듈이고, 데이터 접근 클래스는 low level 모듈이다.
  - DIP에 의해 high level 모듈인 비지니스 로직 클래스가 메소드들을 구현할 때 low level 모듈인 데이터 로직 클래스와 직접적인 의존 관계를 맺는 것은 지양해야 된다.
- Abstraction
  - 비지니스 로직 클래스와 데이터 접근 클래스는 concrete 클래스
  - abstraction은 concrete 클래스가 아닌 추상 클래스나 인터페이스를 말한다.
- 데이터 접근 클래스의 인터페이스
  - 데이터 접근 클래스 내에 존재하는 메소드를 인터페이스로 만들고, concrete 클래스인 데이터 접근 클래스는 이를 상속받아 오버라이딩하는 방식으로 메소드를 구현한다.
  - 데이터 접근 클래스의 생성 역할을 하는 팩토리 클래스에서 반환 타입을 데이터 접근 인터페이스로 하여 concrete 클래스가 아닌 abstraction 클래스를 반환하도록 한다. (하지만 실제로 return되는 것은 부모 클래스 참조 변수에 담긴 자식 클래스 concrete 데이터 접근 클래스이다.)
  - 비지니스 로직 클래스에서는 반환 받은 인터페이스로 메소드에 접근하여 다형성을 통해 비지니스 로직을 구현한다. 

- DIP IoC를 따라 loose coupling 형태로 구현하면 비지니스 로직은 인터페이스를 반환 받기 때문에 데이터 로직 클래스의 이름이 변경되더라도 비지니스 로직 클래스를 변경할 필요가 없으며, 만약 데이터베이스 시스템을 바꾼다고 하더라도, 다른 데이터 베이스 시스템을 사용하는 데이터 접근 클래스가 인터페이스를 상속 받아 메소드를 구현하는 한 비지니스 로직을 변경할 필요가 없다.
